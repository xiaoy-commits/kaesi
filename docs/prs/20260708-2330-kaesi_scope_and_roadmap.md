## 2. 研究目标

回答以下问题，作为后续 Implementation PR 拆分的依据：

1. **运行形态**：脚本目标平台是 Android 模拟器（PC 端）、原生 Android、还是 iOS？
   - MaaFramework 全部支持，但资源模板、OCR 模型、性能基线差异显著。
2. **合规边界**：游戏方用户协议 / 反作弊策略对该类辅助脚本的态度？
   - 若 EULA 禁止，本项目应**自用 / 学习**范围内，不发布，不分发。
3. **MAA 资源来源**：
   - 走 MaaFramework 通用接口（自维护 `resource/`）。
   - 还是复用现有第三方资源（暂未发现「卡厄思梦境」现成 MAA 项目；需确认）。
4. **入口划分**：14 项是否全部纳入 MVP？哪些是 V1 必须，哪些是 V2+？
5. **GUI 形态**：是否需要桌面 UI（参考 `UniversalMaaActuator` 的 Qt6 方案），还是先 CLI/Python 入口？
6. **Python 与 MAA 边界**：业务逻辑写 Python，MAA 只负责图像识别 + 原生交互？

## 4. 目标表现（V1 草案，待用户裁剪）

完成后应满足：

- 一条命令可启动脚本，自动按任务序列执行：
  1. 进入游戏 → 2. 领奖 → 3. 喝咖啡 → 4. 疗愈中心 → 5. 政策审阅 → 6. 共度时光 → 7. 梦讯红点清除 → 8. 教学处自动清理红点 → 9. 模拟 / 出击（按配置）→ 10. 自动推图 → 11. 自动剧情（按配置）。
- 单条任务也可独立运行（CLI 子命令）。
- 每个动作可在 `resource/` 中独立替换模板，无需改代码。
- 失败可重试 / 可跳过，不静默吞错。
- 默认只在本机运行，不内置分发能力。

## 5. 验收边界（V1）

- 默认路径行为不变：脚本不主动修改用户账号数据。
- 每个任务必须有可重复的最小回归用例（截图 + 期望落点）。
- 静默失败禁止：每次动作失败必须日志可见、可追溯。
- 不引入生产级风控对抗（按键精灵式硬件模拟、外挂 hook 等）。
- 不内置云端分发、上传、远控。

## 6. 暂不纳入

- 卡厄思刷开局（涉及账号批量 / 风控风险，V1 不做）。
- GUI（V2 再考虑）。
- 多账号 / 多开管理。
- 跨设备同步。
