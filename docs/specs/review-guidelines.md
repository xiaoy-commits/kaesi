# Review guidelines

这是给 reviewer 的判定规则，不是实现说明。
请必须要严格遵循该规范

## 代码审查原则

**先看边界与验证**
- Don't log PII.
- 先看目标、范围、非目标和验证。
- PR 中的声明、评论和结论不能替代验证。
- 偏好、风格或另一种同样正确的实现，不默认当作缺陷。
- 没有验证的问题都是假设；没有证据的问题，只能作为待核实信息。

**范围与复杂度**
- 报告超出用户请求、超出 plan 边界或明显过度复杂的改动。
- 如果写了 200 行，而问题可以用 50 行清晰解决?
- 不要让谷歌的L9工程师认为这段实现过度复杂或者需求或方案过度设计。
- 避免过度工程化。只进行被直接要求的修改，或明显必要的修改。保持方案简单、聚焦。
- 新增实现应优先复用仓库既有抽象或成熟、与当前场景匹配的轻量库；不要把可直接复用的问题变成重复造轮子。
- 面对几乎不可能发生的问题，不应该进行冗余报告。

**行为与验证**
- 每一处改动都应能直接追溯到用户请求。
- 问题应以代码、测试、运行行为、接口契约或可复现路径为依据。
- 改动有充分的test覆盖，并且测试用例清晰表达了预期行为。
- test行为符合PR预期,而非test行为符合代码预期
- 非必要隐式行为不允许
- 特定于项目的隐式行为需要清晰注释说明
- 静默失败是不允许的。

**代码质量**
- 代码质量判断以 KISS / YAGNI / DRY / SOLID 为隐含原则，但只在它们明确影响正确性、边界、一致性或维护性时作为审查问题提出。
- 遵循了仓库的约定的风格和规范。
- 项目不能存在demo风格的代码。
- 不能将开发语义和生产语义混淆，不能将开发语义暴露到用户界面。
- 不要为不可能发生的场景添加错误处理、fallback 或校验。信任内部代码和 framework guarantees。只在系统边界进行校验，例如 user input、external APIs。能直接修改代码时，不要使用 feature flags 或 backwards-compatibility shims。
- 不要为一次性操作创建 helpers、utilities 或 abstractions。不要为假设中的未来需求设计。合适的复杂度是当前任务所需的最小复杂度；三行相似代码优于过早抽象。
- 避免 backwards-compatibility hacks，例如重命名未使用的 `_vars`、重新导出 types、为已删除代码添加 `// removed` 注释等。如果确定某个内容未被使用，可以直接彻底删除。
- DEV 阶段，不为了过去的旧代码，旧数据库等开发阶段有关的内容书写有关兼容代码. 不包含项目演进过程的兼容代码.

## PR 契合度

代码本身正确，不代表 PR 适合当前项目。审查时必须检查以下六个对齐：

* **目标是否对齐**：是否解决了本次真正要解决的问题，是否夹带顺手优化或额外功能。
* **边界是否对齐**：是否越过用户请求、Issue、plan 或当前 PR 范围。
* **架构是否对齐**：是否复用项目已有抽象、目录结构、状态管理、接口封装和错误处理方式，是否另起一套体系。
* **产品语义是否对齐**：用户可见行为是否符合项目当前产品逻辑，而不是仅仅技术上成立。
* **运行环境是否对齐**：是否只在本地、mock 或 demo 环境成立，真实环境、小程序端、test-prod 或生产配置是否验证过。
* **维护成本是否对齐**：是否为了一个小问题引入长期复杂度，例如额外 helper、adapter、feature flag、兼容层、配置项或过早抽象。

如果 PR 能跑但不符合上述对齐，应作为审查问题提出。
